## Critical Collaboration Rule

The AI must actively challenge gameplay, balance, and architecture ideas. Do not merely agree.

For every major proposed mechanic or architecture shift, check for:

- logical holes
- easy bypasses and exploits
- dominant strategies
- burst synchronization abuse
- defensive-trigger baiting
- late-game saturation
- early-game uselessness
- hidden order-of-operations surprises
- "blocks everything or blocks nothing" behavior
- loss of player-facing readability
- hot-path performance regressions
- architectural drift away from the current modular-monolith direction

If a proposal conflicts with these critical decisions or with an archived/past rejection, call it out clearly. This is one of the main values of AI collaboration in this project.

## Solo-Indie Balance Rule

The game must favor systems that are understandable, predictable, balanceable, and testable without a large QA team.

Prefer mechanics that:

- degrade gracefully instead of snapping between total protection and instant death
- can be steered through clear parameters
- are easy to simulate deterministically
- expose cause/effect to the player
- avoid mandatory hidden caps/defaults such as "cap all resistances"

Future balance work should use deterministic simulations/tests for survivability, burst handling, sustained damage, trigger consumption, guard consumption, and dominant-strategy detection.

## Game Vision

The game is a 2D single-player autobattler / build-optimization game inspired by Backpack Battles and ARPG itemization such as Path of Exile.

The core gameplay is arranging and configuring items in an inventory/grid/gear system so the player optimizes attack, defense, resource flow, and interactions.

The player should feel overloaded by build possibilities, not by hidden rules.

Critical readability goal:

- The player should not need a wiki to understand mechanics.
- Complexity should come from many visible ways to influence the system.
- Rules should be readable, consistent, and inspectable.
- Player agency and clear cause/effect are extremely important.

Important game-feel goal:

- The player should feel they are constructing the logic of the game's attack and defense mechanisms.
- Attack/defense should be sandbox-like and configurable through build choices.
- There should not be one immutable attack system or one immutable defense system.

## Architecture Direction

Prefer the existing modular-monolith style:

- domain logic split by feature modules such as `Flow`, `CombatContext`, `Character`, `Inventory`, `Item`, and related contracts
- public contracts and APIs at module boundaries
- internal domain implementations where practical
- dependency composition through Zenject installers
- UI separated from domain/gameplay logic

Public APIs should describe domain/gameplay stages, not technical implementation details.

Avoid new architectural layers unless they remove real complexity, reduce meaningful duplication, or clearly match an established local pattern.

## Object Invariants And Guards

Use `NullGuard` helpers when constructing domain/gameplay objects.

Required constructor parameters should fail fast instead of being silently normalized. Do not hide invalid input with fallbacks such as `value ?? string.Empty` unless the fallback is itself the explicitly documented domain value.

When a parameter is optional, make that optionality explicit in the object invariant. Example: `CombatHook.None` may have no observed combat event type, but concrete hooks must declare and validate the observed `CombatEventType`.

## Commands, Events, Hooks

Current meaning:

- `CombatCommand` is an intent/request.
- `CombatEvent` is a synchronous domain fact.
- `CombatHook` is a reaction/hook point to facts.
- `CombatHook` is not a second kind of `CombatEvent`. It is a separate game-mechanic reaction identity, often visible to the player.

Terms such as `OnAttack` and `OnIncomingAttackDamage` are game-mechanic hook points, not C# events, async work, threads, or technical pub/sub.

Concrete event/fact classes should avoid `On...` names. `On...` remains suitable for hook/player-facing trigger language.

Implemented terminology guard:

- domain facts implement `ICombatEvent`
- hook definitions implement `ICombatHook`
- event-triggered entry points must declare a concrete `CombatHook`
- combat-tick entry points should use `CombatHook.none()`
- `CombatHook.onIncomingAttackDamage()` observes `CombatEventType.INCOMING_ATTACK_DAMAGE`
- `IncomingAttackDamageCombatEvent` remains the fact name; `OnIncomingAttackDamage` remains the hook/player-facing phrase

Avoid exploding into separate processor interfaces per event type. A generic staged reaction workflow is preferred where needed:

`start(item, combatEvent) -> match() -> validateProcessability() -> prepareExecution() -> react()`

Important distinction:

- `match()` means "does this item react to this event type/domain situation?"
- `validateProcessability()` means "can it react right now?" including cooldown/resources/capacity.

Do not mix match logic with cooldown/resource/processability checks.

## Combat Runtime And Tick

Combat tick is a hot path.

Performance assumptions:

- combat tick may run around 10 times per second and potentially more later
- target scale can reach roughly 1,000-2,000 active items and flows
- some smoke tests use 5,000 active flows/items as catastrophic regression guards

Hot-path rules:

- avoid per-tick allocations
- avoid LINQ materialization in tick phases
- avoid reflection
- avoid broad collection rebuilding
- avoid hidden O(n*m) scans
- prefer stable indexes, reusable buffers, explicit phases, cheap primitive queries, and bounded work budgets

`ICombatQueries` should remain lightweight and return simple values such as `int`, `float`, `bool`, `string`, enums, or cheap primitive/value objects. Do not expose mutable runtime objects, heavy DTO graphs, or complex collections from query surfaces.

## Flow Runtime

The project moved away from fire-and-forget async flow processing.

Current direction:

- active flow lifecycle is owned by combat runtime/characters, not by entry points
- flows are ticked runtime state
- active flows are stored, ticked, and removed after completion
- combat/character runtime should use stable snapshots/reusable buffers for tick phases

Open architectural caution:

- global combat tick phase order should not be buried ad hoc inside `CombatCharacter`
- character can own local runtime state, but combat-level tick order is a combat runtime rule

## Time And Cast Semantics

`CombatTicks` is the canonical gameplay time unit for now.

Do not introduce final seconds-to-ticks semantics unless needed deliberately.

Runtime battle speed control is a battle-loop scheduling concern:

- speed multipliers change how quickly real elapsed time produces combat ticks
- pause stops producing combat ticks and must not accumulate delayed ticks to replay on resume
- speed and pause must not multiply item power, cast ticks, trigger intervals, damage, guard power, or other domain values
- battle UI may use unscaled Unity frame time to drive this scheduler, while combat/domain logic continues to receive explicit `CombatTicks`

Item cast time:

- `ItemCastTime` is a domain value object
- item cast duration is based on ticks
- base item cast time means cost to process one cell
- actual cast duration for a flow's assigned row is `ItemCastTime * slotCellCount`

Normal item processing currently uses cast time, not cooldown/recovery.

Entry point trigger timing is a separate concept and may later need its own UI countdown/charge presentation.

Damage packet timing after 2026-06-27:

- offensive flow output no longer applies incoming damage immediately
- attack output creates a ticked `ActiveDamagePacket`
- packet timing is combat-domain timing, not only UI animation; offscreen characters process the same packet timing as visible characters
- current default packet stage duration is 10 combat ticks per stage, matching the current normal speed assumption of about 10 ticks per real second
- the duration is configurable through `DamagePacketProcessingSettings`
- current packet order is Travel -> IncomingHit -> Stability -> Guard -> HP
- `IncomingHit` dispatches `IncomingAttackDamageCombatEvent`, so defensive event flows can still intercept the packet before it becomes resolved layer damage
- if no defensive flow handles the incoming hit, or once defensive flow output resolves damage, a resolved packet proceeds through Stability -> Guard -> HP over ticks
- `DamagePacketLayerProcessedDtoEvent.completesPacket` is the domain/presentation fact that a visual packet value should finish; UI must not infer completion only from `DamagePacketLayer.Hp`
- `IncomingHit` completes the pre-defense incoming packet value, while HP or any future fully absorbing layer completes the resolved layer packet value
- `ICombatQueries.getActiveDamagePacketCount()` is a primitive query intended for tests/debug assertions that packet queues are fully drained after ticked processing
- UI may visualize packet layer facts, but domain packet processing must remain deterministic and independent from visible windows, DOTween, frame rate, or active selection
- damage packet UI should present a single falling damage value moving through the visible layer stack rather than debug text such as `incoming>outgoing`; after each layer, the displayed value is the value continuing to the next layer
- the selected-character defense overlay may show only the latest 5 visible falling packet values; if more packet values are active, the UI shows a `5+` overflow marker and keeps combat-domain packet processing unchanged
- even though `IncomingHit` completes the pre-defense domain packet, selected-character UI should hold that incoming value and visually continue it into the next resolved layer packet when Stability/Guard/HP processing begins, so the player does not see the number disappear between stages

## Inventory, Items, Shape, And Processing Slots

Inventory/item shape is gameplay-relevant.

Current model:

- every occupied shape cell is treated as active
- each local shape row is one flow processing slot
- `CombatPlacedItemRuntimeState` tracks per-item flow capacity/current active flow count
- processing capacity should not be counted by scanning all active flows in the tick hot path

Future direction:

- item instances may distinguish base shape cells from active/unlocked cells
- low-rarity items may have only some active cells
- cell-level components/overlays may modify row cast time, effects, damage type, or slot behavior

Do not add full per-cell/component logic until the base row-slot model is stable.

## Flow Visualization And UI Boundary

Domain/runtime must not own presentation visual identity.

Domain may expose neutral identity/state:

- `ActiveFlowId`
- flow kind
- ordered processing path
- current processing slot
- cast state
- future gameplay/statistics data

UI/presentation maps these to colors, opacity, dash phase, marker glyphs, highlight state, focus state, and palettes.

Current UI direction:

- default inventory flow rendering should be conservative
- flow path lines are hidden in the unfocused default inventory view
- flow path visualization appears after focusing/selecting an item, entry point, or visible flow
- item visual states should be centralized: normal, dimmed, focused, related, stale-related
- dimming should use opacity/desaturation/darkening; strong outline/glow reserved for selected/focused objects

Entry point UI color convention:

- `FlowKind.Damage` / offensive entry points use warm orange
- `FlowKind.Defense` / defensive entry points use green/turquoise
- this is UI-only

## Game Events And Modifiers

Future passive modifiers may apply at:

- whole-game layer
- character layer
- inventory/module layer
- later item/cell/component layers

Event-aware modifiers are central to the planned game. Examples:

- on attack
- on incoming attack damage
- every N incoming damage events
- conditional reactions based on combat counters

Combat/character runtime should record live gameplay event counters because mechanics may read them during combat.

Performance rule:

- do not scan all items/modifiers for every frequent event
- prepare indexes of reaction candidates by hook/event kind and owner scope
- use cheap counter increments, reusable buffers, and primitive query APIs

Inventory/module-level event counters remain future work unless a concrete mechanic needs them.

## Defensive Incoming Damage Foundation

Current implemented direction:

- incoming attack damage can trigger an event-triggered defensive entry point if available/free
- the created flow is `FlowKind.Defense`
- in defensive flow, `DamageRole.ATTACK` means incoming damage being processed before final HP application
- defensive flow uses the same general flow/item mechanics as offensive flow
- `IncomingAttackDamageCombatEvent` is the concrete combat fact
- `OnIncomingAttackDamage` is the future hook/player-facing phrase

Anti-loop rule:

- once incoming damage has been processed by defensive flow, resulting final damage is applied directly
- it must not be registered again as fresh incoming attack damage
- it must not trigger another defensive flow through the same damage

Open direction:

- decide whether `ResolvedDamage` is final naming
- define which `DamageRole` values are legal in each `FlowKind`
- counter/spike damage from defensive flow must use a distinct role such as `SPIKE_DAMAGE`, not normal incoming attack damage
- current event-triggered defensive entry point selection uses a round-robin cursor among registered defensive event entry points; this is a current implementation detail and optimization, not yet confirmed as the final player-facing gameplay rule
- revisit whether defensive event entry points should use round-robin, strict inventory/priority order, explicit priority, conditional matching, or another readable selection model

## Defense Design: Guard, Barrier, And Readability

This area is active design exploration, not final architecture.

Rejected/risky directions:

- pure flat damage reduction is difficult to balance across early/late game
- pure percent reduction risks late-game "100% mitigation/default cap" behavior
- static resistance percentages risk recreating ARPG "cap all resists" meta
- naive "all damage in one tick becomes one packet" does not solve synchronization across multi-second windows
- turning all damage into delayed pressure/DoT may not fit an autobattler where player decisions are prebuilt, not manual real-time reactions

Promising direction:

- defensive flow may generate one defense/guard portion rather than immediately reducing incoming damage
- guard portions can have strength/rating, conditions, tags, priority, durability/strain, expiry, and storage limits
- rapid small attacks may become guard-breaking / guard-draining tools
- heavy guards may answer large hits but ignore/avoid being consumed by small hits
- typed wards may answer specific damage types
- healing entry points are a future idea, not yet specified
- event entry points reacting to direct damage intake remain possible but must avoid recursion, baiting, and unclear timing

Critical risk:

- direct numeric/flat barrier portions may still collapse into binary outcomes: full protection or instant death
- the defense model should fail gracefully

Current implemented guard foundation after 2026-06-06:

- flow output now has a separate `GuardPower` result alongside `attackPower`
- `GuardPower` is not a `DamageRole`; it must stay separate because `DamageRole` describes damage-stream roles
- `AddGuardPower` lets entry points and normal items add guard power to the current flow result
- when a flow is consumed, positive `GuardPower` creates one prepared guard portion on the flow owner before flow-specific attack/damage handling
- prepared guards are combat runtime state owned by `CombatCharacter`, not inventory state and not base HP data
- incoming `ResolvedDamage` is reduced by prepared guards before HP is changed
- each prepared guard portion can react at most once to one damage packet
- using a guard strains it by the blocked damage amount; empty guards are removed
- current generic guard formula is `blocked = damage * (1 - (damage / (damage + guardPower))^2)`, rounded down and clamped so positive incoming damage is never fully blocked
- current readable storage limit is 30 prepared guard portions per character
- when the 31st+ guard would be created, the oldest guard is replaced by the new guard; this is intentionally a readable queue rule, not a hidden weakest-merge rule
- guard creation and attack creation publish neutral combat-context DTO events carrying the final flow processing slot so UI can draw a beam from the last processed item without domain depending on views
- guard absorption publishes a neutral character DTO event with incoming damage, blocked damage, remaining damage, destroyed guard count, and remaining guard power
- current catalog `DefenseEntryPointGem` and `Shield` demonstrate guard generation instead of flat `ATTACK` reduction, currently at attack-scale values of 2 and 5 guard power
- character-level prepared guard creation exposes only the result-bearing `tryAddGuardPower(...)` command; avoid parallel fire-and-forget APIs when the result carries domain facts needed by UI or tests

Open guard cautions:

- the first implementation is generic and untyped: no guard tags, damage-type matching, expiry, priority, or minimum-trigger conditions yet
- the 30-portion cap and oldest-replacement overflow are provisional balance parameters
- heavy guards that should ignore small bait hits still need explicit conditions/tags later
- future deterministic balance simulations must challenge burst, rapid-small-hit drain, guard overflow, baiting, and late-game guard stacking

## Defense Layer Order And Stability

This area is active design exploration, but the first implementation now exists.

Current design direction after 2026-06-22:

- the global armor/resistance-like layer is currently named `Stability` in code and should be presented to the player as `Stabilnosc`/`Stabilność` unless later naming work changes it
- each battle currently starts with baseline Stability 100 as the readable early-game reference point
- baseline Stability is the size/reference of the Stability bar, not a direct percent cap; late-game characters may have a much larger baseline while keeping the same standard mitigation at full bar
- defensive flows may generate more Stability during combat, so combat becomes a pressure test of whether the build can maintain its defensive baseline while attacks strain it
- Stability should be one readable global defensive number, not a hidden ARPG-style resistance cap
- increasing it above the baseline should be rewarded, but with diminishing percentage effectiveness and with pressure/counterplay from armor strain, pierce, break, or ignore-style attack properties
- focusing only on Stability should not be the automatically best defense because attacks can be designed to drain, pierce, ignore, or break this layer
- current mitigation formula is `damageAfter = incomingDamage * baselineStability / (baselineStability + currentStability)`, clamped so positive incoming damage still deals at least 1 damage
- full baseline Stability therefore gives about 50% reduction whether the baseline is 100 in early game or 1000+ in late game
- ordinary incoming attacks currently strain Stability by roughly one third of their incoming damage and clamp ordinary strain at 0; negative Stability remains a future vulnerability/counterplay concept, not something ordinary attacks create by default
- Stability can exceed baseline, but over-baseline Stability decays over combat ticks: no decay up to about 110% baseline, then very slow fractional decay that accelerates smoothly as Stability moves farther above the threshold
- the current over-baseline decay curve is intentionally steerable character/balance space; the current prototype uses a convex "opposite of logarithmic" curve, so near-threshold overcap lasts a while, while large overcap around double the threshold and beyond decays much faster
- previous incoming damage flow was prepared Guard -> HP; current flow is Stability -> prepared Guard -> HP

Layer presentation direction:

- the inventory/combat UI shows defense layers as ordered rows above the item grid
- the top row is processed first; lower rows are processed afterward
- near-term default order is Stability first, prepared Guard second, HP last
- each defense layer should read as its own component/section in the stack
- falling damage packet values should live in a separate left-side packet lane component; Stability, Guard, and HP layer components are laid out to the right of that lane
- the damage-process lane must be wide enough for five packet values side by side with spacing; do not compress five packets into a narrow single-column lane
- the damage-process lane should communicate stages through simple aligned bands/separators rather than arrow glyphs; arrows in mockups were only directional annotation, not desired UI chrome
- layer overlays must be printable for a selected character even when the character has no item views yet; presentation should attach overlays to the item-layer transform, not infer the parent only from existing item instances
- current BattleScene should keep the separated GUI layout with character panel, item/stash buttons, defense process, and item grid visible together; do not stretch the item grid/inventory panel to the full screen until the whole GUI layout is redesigned around that
- fullscreen/background UI panel images must not keep `RaycastTarget` enabled unless they intentionally handle pointer input; otherwise they create invisible click-blocking layers over buttons/items
- HP uses a normal fill bar: full HP is a full bar, half HP is a half bar
- HP shows current/max HP centered on the HP bar
- Stability should be a more prominent/taller component than the compact rows, shows current/baseline as the readable value, fills against baseline, and separately shows `Damage Reduction: X%`
- Stability `Damage Reduction` color should communicate state through a readable gradient: negative values move toward very dark black/burgundy/red, low positive reduction is red, mid-low reduction uses muted amber/orange rather than bright yellow, normal around 45-55% is blue, and high reduction moves toward green
- near-term defense-layer UI palette is dark fantasy only for these layer components: main background around `#100C17`, panel `#1C1426`, raised panel `#291C38`, border `#554165`, old-gold accent `#C49A52`, main text `#E9E2EE`, secondary text `#B1A5B7`, defense blue `#4F78A8`, healing green `#639775`, warning amber `#D18B45`, power red `#C6535D`, critical red `#A93644`
- the Guard layer should remain visible as a layer section even when there are no prepared guards, so the processing order does not disappear
- a later equipment/editing feature may let the player configure the order of defensive layers, for example armor before guard or guard before armor
- row order must be a visible player-facing rule, not hidden combat pipeline trivia
- a future barrier layer is planned as generated extra HP or a related buffer, so "Barrier" should not be used as the name for the Stability layer

## Guard Power Readability

Candidate player-facing names:

- Guard Power
- Guard Strength
- Shield Strength
- Ward Strength
- Protection
- Stability
- Polish candidates: `Siła Gardy`, `Siła Osłony`, `Moc Osłony`, `Stabilność`

Current favorite direction:

- guard/defensive ability can be a single main value if UI translates it into practical examples
- tooltip should show concrete behavior, not just a raw rating

Example:

```text
Guard Power: 120
Against 60 damage: blocks ~53
Against 120 damage: blocks ~90
Against 240 damage: blocks ~133
```

This lets players understand behavior without memorizing a formula.

Critical abstract-stat readability rule:

- one abstract stat can be manageable if the opposing side remains concrete
- two abstract opposing stats quickly destroy readability

Therefore:

- abstract-ish `Guard Power` may work when shown against concrete incoming damage values
- avoid abstract `Guard Power` opposed to abstract `Offensive Power`
- attack flow should produce concrete damage-like values whenever possible

If attack output becomes a generic abstract offensive rating, guard tooltips lose context and the game risks becoming opaque ARPG-style offensive/defensive stat soup.

Current implemented UI direction after 2026-06-06:

- prepared guards are shown above the inventory area as compact shield-like guard icons
- each icon shows a short 3-4 character-ish guard power label such as `999`, `2.5k`, or `1m`
- the displayed guard list is limited by inventory width and up to two rows
- hovering a guard shows concrete examples of how much damage it blocks against smaller/matching/larger incoming damage
- guard creation, attack creation, and guard absorption are visualized in UI from domain DTO facts; the domain must not know about beams, popups, colors, glyphs, or inventory transforms
- the current icon is a free/runtime UI glyph approach, not a paid asset dependency; it can be replaced later by a prefab or free icon asset

## Flow Ports And Output Commit

Current implemented direction after 2026-06-07:

- ports are metadata/characteristics visible on items, not a new UI-only concept
- current `IEntryPointDefinition` still exists and must not be removed yet
- current port implementation is an adapter over the existing entry-point/flow-kind contracts; it is not the final fully neutral port architecture
- a port should not be an item-processing/cast rules engine; cast/effect transformations remain item/action behavior
- output ports are commit points for accumulated flow power
- when a flow reaches an output port, the output port itself is not cast and does not execute its own item effects
- attack and guard DTO facts should originate from the output/final processing slot so UI can draw beams from the watched output location
- port-aware input flows that do not reach an output are discarded and reported as no-output/wasted facts
- legacy entry-point flows that do not reach an output still commit for compatibility; legacy no-output warnings are published only in the developer runtime profile because emitting them for every legacy flow is too expensive in hot-path production stress scenarios
- UI may show `IN`, `OUT`, no-output, and output-reached signals, but domain rules must remain in flow/combat modules

Important caveat:

- because the current entry-point contract still requires `FlowKind`, example input ports may temporarily expose a damage/defense flow kind even though the target design is for input ports to answer mostly "when/how triggered", while output/flow contents decide what is committed
- future port-aware defensive inputs need an explicit no-output policy; a defensive flow that consumes an incoming-damage event and then discards without output must not accidentally erase the original damage unless that is an intentional, visible mechanic

## Prepared Guard Portion Queue

Current implemented direction after 2026-06-07:

- previous mechanism: prepared guards had a 20-portion cap, overflow merged into the weakest guard at 70% efficiency, and incoming damage selected stronger guards first
- new mechanism: prepared guards are a readable bounded queue with max 30 active portions
- new guards are appended to the right/end of the queue
- when all 30 slots are occupied, adding a new guard replaces the oldest guard
- incoming damage consumes prepared guards from left to right
- one damage packet may continue into later guards if earlier guards do not absorb all remaining damage
- replacement and absorption should be reported as domain facts/presentation DTO data so UI can show a push/replacement and shield-splitting/blocked amount effect

Design reason:

- the old weakest-merge/strongest-first behavior was a hidden global rule that players would have to infer
- a left-to-right bounded queue is easier to read, easier to visualize, and keeps the per-hit/per-add guard work bounded by 30
- balance should move toward item/output properties deciding how guard portions are created/replaced/split, rather than more global guard-selection rules

## Static Gear Segment

This area is active design exploration, not final architecture.

User is considering a separate static gear/loadout segment outside the flow grid.

Possible naming:

- static gear
- loadout
- modules
- core equipment
- relics
- frame

Purpose should not be "hidden resistance caps".

Better direction:

- define baseline economy and constraints
- max HP
- max barrier/guard storage
- guard generation/recharge
- max prepared guard portions
- overload efficiency
- attack/resource generation

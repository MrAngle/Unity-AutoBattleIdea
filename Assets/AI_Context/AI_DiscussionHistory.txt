AI 2026-05-26 - Combat runtime context and ticked flow

User wants CombatContext to carry live combat-wide runtime information, not only static data. This includes participating characters, available actions, active flow counts, created flow counters, and similar state used by item conditions.

Performance assumption: combat tick may run about 10 times per second, and possibly more often later. The target scale can reach roughly 1,000-2,000 active items and flows. Anything in tick must be treated as hot-path code. Avoid per-tick allocations, LINQ materialization, reflection, broad collection rebuilds, and hidden O(n*m) behavior. Prefer indexes, reusable buffers, explicit phase plans, cheap query endpoints, and bounded work budgets.

Important decision: ICombatQueries must remain a lightweight query surface. It should return simple values such as int, float, bool, string, enums, or cheap primitive-like values. It must not expose complex runtime objects, collections, wrappers, DTO graphs, or mutable domain internals.

Current concrete need: item conditions should be able to ask whether there are more than 3 active flows and decide not to create a new flow. This first needs live active flow tracking and query endpoints.

Direction for implementation: stop treating FlowProcessor as fire-and-forget async. Flow should become ticked runtime state, similar in spirit to InventoryPlacedEntryPoint ticking. Active flows should be stored by combat runtime/characters, ticked during combat ticks, and removed only after completion.

Implementation detail refined: combat tick should process phases from stable snapshots. Active flows are copied as references into a reusable buffer, ticked from that buffer, and then the active flow list is compacted. This avoids newly added flows affecting the current flow phase while also avoiding per-tick list allocation. For now, flow phase runs before item/entry-point phase, so item conditions see state after already-active flows were processed and before newly created flows are processed on a later tick.

Follow-up refinement: snapshot buffers should not live as vague fields like flowTickBuffer on CombatCharacter. Tick orchestration should be explicit as a small tick plan/action chain: first include participants for the tick, then execute named phases in order. CombatCharacter can own active runtime state, but the order and snapshot mechanics belong to CombatTickPlan.

Architectural concern: the order of combat tick phases should not be defined ad hoc inside CombatCharacter. CombatCharacter can expose or own character-local runtime state, but the global order of phases is a combat runtime rule. The next refactor should move phase ordering to a combat-level orchestrator/runner or a reusable tick pipeline, while Character supplies participants/state.

Immediate code refinement: CombatCharacter may own a CombatTickPlan-like helper, but CombatCharacter should not spell out the phase order through a chain. It should pass active flows, tickable items, character id, combat ticks, and capabilities into the plan/runner. The plan/runner owns the phase order.

IDE/performance refinement: methods that execute tick phases should not look like cheap builder calls. CombatTickPlan should separate preparation of snapshots from execution of prepared phases. Expensive phase methods should be named as execution methods, not fluent "include/build" methods.

Open point for later: decide how Duration seconds map to CombatTicks. For the first implementation step, avoid locking in a final time conversion unless needed.

Previous flow behavior before today's refactor:

- FlowProcessor exposed start() and internally launched startAsync() as fire-and-forget work.
- stepLoopAsync processed nodes asynchronously until there were no more nodes.
- For every current node, FlowProcessor prepared ExecuteActionCommand and delegated execution to IActionExecutor.
- ActionExecutorService created PreparedAction, read IActionDescription.getCastTime(), waited with TimeModule.ContinueIn(duration), and only then executed the item effects.
- The visible delay between items came mostly from each item's Duration/castTime through ActionExecutorService. Router logic only chose the next item; it did not own the delay.
- FlowProcessorSettings.maxStepsPerSlice and IFlowStepScheduler were an additional async/yield safeguard, not the main item-to-item delay.

What changed today:

- FlowProcessor no longer exposes start() and no longer owns async fire-and-forget execution.
- IFlowProcessor now exposes tick(CombatTicks) and isFinished().
- FlowProcessor applies node effects during tick and advances through routed nodes in the ticked runtime model.
- Active flows are stored on CombatCharacter for this first implementation step, counted through simple query endpoints, and removed after completion.
- CombatCharacter now reports active and created flow counts through ICharacterCombatQueries, and combat-wide ICombatQueries aggregates those counts.
- IFlowStepScheduler was removed because the current direction is ticked runtime rather than async/yielded flow processing.
- CombatTickPlan was introduced as an explicit phase plan. CombatCharacter passes participants/context to it, while the plan owns the local phase order for now.

Important consequence:

- The old castTime-based item delay is not yet represented in the new ticked model. Duration-to-CombatTicks mapping and per-flow/per-character tick work budgeting still need a deliberate design step.

AI 2026-05-28 - Cast time, tick semantics, validation, and random item value notes

User wants to restore/support item castTime after the previous tick refactor, while keeping combat tick performance as a critical hot path. Current concern: using raw ticks everywhere is mechanically clean but might make item behavior harder to reason about in user-friendly terms; using a separate "GameSecond" might improve readability but risks creating a second competing gameplay time unit.

Current recommendation: keep `CombatTicks` as the canonical domain/gameplay time unit and do not introduce `GameSecond` as a separate domain concept yet. If user-facing seconds are needed, derive them at UI/config boundaries from `BattleSessionSettings`. For item characteristics, prefer tick-based value objects: either `CombatTicks` directly or a small semantic wrapper such as `ItemCastTime`/`CastTime` that contains `CombatTicks` and validates invariants. Avoid `Duration(float)` for gameplay mechanics; floats should stay at presentation/Unity animation/time-boundary layers unless wrapped by a domain time object.

Cast time direction under ticked flow: flow should not process every node immediately up to `maxStepsPerSlice` as normal pacing. Each active flow should keep per-flow progress for the current node's cast time, accumulate elapsed `CombatTicks`, execute the item only when enough ticks were accumulated, then route to the next node and reset progress for that next node. `FlowProcessorSettings.maxStepsPerSlice` remains only an emergency safety cap against runaway graph walking in one tick.

User also asked to record a broader architectural rule: every object should validate data consistency. Null guards are the main current form, but the real principle is invariant validation. Domain/config/runtime objects should check nulls, ids, enum values, positive/non-negative quantities, dimensions, and coherent combinations at construction/update boundaries. Avoid putting expensive validation in hot tick loops; tick code should receive already coherent data.

Design note for later item generation/balancing: user strongly wants highly generic random items, where two identical items are extremely rare and items can have many different specifications in many orders. This makes item valuation one of the most interesting and dangerous design areas. The value model must reason contextually, not just by raw numbers. Example: very long cast time may be valuable for defensive items, while very short cast time is likely highly valuable for offensive items. This is not being solved now, but future item characteristics should be designed so contextual valuation can be added.

AI 2026-05-28 - Implemented tick-owned item cast time

Previous mechanism: item action descriptions exposed `Duration(float)`, `ActionExecutorService` read that float and waited through `TimeModule.ContinueIn(...)`, and old async flow processing delegated timing to the executor. After the earlier flow refactor this mechanism was no longer active in the ticked model, so `FlowProcessor.tick(...)` processed nodes immediately up to `FlowProcessorSettings.maxStepsPerSlice`.

New mechanism: `Duration(float)` was removed from gameplay/domain action descriptions. `IActionDescription.getCastTime()` now returns `ItemCastTime`, a small value object that wraps `CombatTicks` and rejects negative cast time. Item catalog definitions now express cast time directly in ticks. `FlowProcessor` stores `ticksUntilCurrentNodeExecution` for the current node, spends incoming `CombatTicks` on each flow tick, executes the current node only when the cast reaches zero, then routes to the next node and reads that next item's cast time once. Instant cast nodes can still execute within the same tick, but the existing `maxStepsPerSlice` safety cap bounds runaway instant chains.

Important ownership decision: item owns the cast-time characteristic, but active flow owns runtime progress through that cast time. This keeps item data stable and makes flow lifecycle deterministic without async/freezing/waiting tasks.

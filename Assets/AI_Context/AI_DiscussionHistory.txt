AI 2026-05-26 - Combat runtime context and ticked flow

User wants CombatContext to carry live combat-wide runtime information, not only static data. This includes participating characters, available actions, active flow counts, created flow counters, and similar state used by item conditions.

Performance assumption: combat tick may run about 10 times per second, and possibly more often later. The target scale can reach roughly 1,000-2,000 active items and flows. Anything in tick must be treated as hot-path code. Avoid per-tick allocations, LINQ materialization, reflection, broad collection rebuilds, and hidden O(n*m) behavior. Prefer indexes, reusable buffers, explicit phase plans, cheap query endpoints, and bounded work budgets.

Important decision: ICombatQueries must remain a lightweight query surface. It should return simple values such as int, float, bool, string, enums, or cheap primitive-like values. It must not expose complex runtime objects, collections, wrappers, DTO graphs, or mutable domain internals.

Current concrete need: item conditions should be able to ask whether there are more than 3 active flows and decide not to create a new flow. This first needs live active flow tracking and query endpoints.

Direction for implementation: stop treating FlowProcessor as fire-and-forget async. Flow should become ticked runtime state, similar in spirit to InventoryPlacedEntryPoint ticking. Active flows should be stored by combat runtime/characters, ticked during combat ticks, and removed only after completion.

Implementation detail refined: combat tick should process phases from stable snapshots. Active flows are copied as references into a reusable buffer, ticked from that buffer, and then the active flow list is compacted. This avoids newly added flows affecting the current flow phase while also avoiding per-tick list allocation. For now, flow phase runs before item/entry-point phase, so item conditions see state after already-active flows were processed and before newly created flows are processed on a later tick.

Follow-up refinement: snapshot buffers should not live as vague fields like flowTickBuffer on CombatCharacter. Tick orchestration should be explicit as a small tick plan/action chain: first include participants for the tick, then execute named phases in order. CombatCharacter can own active runtime state, but the order and snapshot mechanics belong to CombatTickPlan.

Architectural concern: the order of combat tick phases should not be defined ad hoc inside CombatCharacter. CombatCharacter can expose or own character-local runtime state, but the global order of phases is a combat runtime rule. The next refactor should move phase ordering to a combat-level orchestrator/runner or a reusable tick pipeline, while Character supplies participants/state.

Immediate code refinement: CombatCharacter may own a CombatTickPlan-like helper, but CombatCharacter should not spell out the phase order through a chain. It should pass active flows, tickable items, character id, combat ticks, and capabilities into the plan/runner. The plan/runner owns the phase order.

IDE/performance refinement: methods that execute tick phases should not look like cheap builder calls. CombatTickPlan should separate preparation of snapshots from execution of prepared phases. Expensive phase methods should be named as execution methods, not fluent "include/build" methods.

Open point for later: decide how Duration seconds map to CombatTicks. For the first implementation step, avoid locking in a final time conversion unless needed.

Previous flow behavior before today's refactor:

- FlowProcessor exposed start() and internally launched startAsync() as fire-and-forget work.
- stepLoopAsync processed nodes asynchronously until there were no more nodes.
- For every current node, FlowProcessor prepared ExecuteActionCommand and delegated execution to IActionExecutor.
- ActionExecutorService created PreparedAction, read IActionDescription.getCastTime(), waited with TimeModule.ContinueIn(duration), and only then executed the item effects.
- The visible delay between items came mostly from each item's Duration/castTime through ActionExecutorService. Router logic only chose the next item; it did not own the delay.
- FlowProcessorSettings.maxStepsPerSlice and IFlowStepScheduler were an additional async/yield safeguard, not the main item-to-item delay.

What changed today:

- FlowProcessor no longer exposes start() and no longer owns async fire-and-forget execution.
- IFlowProcessor now exposes tick(CombatTicks) and isFinished().
- FlowProcessor applies node effects during tick and advances through routed nodes in the ticked runtime model.
- Active flows are stored on CombatCharacter for this first implementation step, counted through simple query endpoints, and removed after completion.
- CombatCharacter now reports active and created flow counts through ICharacterCombatQueries, and combat-wide ICombatQueries aggregates those counts.
- IFlowStepScheduler was removed because the current direction is ticked runtime rather than async/yielded flow processing.
- CombatTickPlan was introduced as an explicit phase plan. CombatCharacter passes participants/context to it, while the plan owns the local phase order for now.

Important consequence:

- The old castTime-based item delay is not yet represented in the new ticked model. Duration-to-CombatTicks mapping and per-flow/per-character tick work budgeting still need a deliberate design step.

AI 2026-05-27 - Combat tick stress smoke test refinement

User clarified that the performance smoke should stress a more complex fight, not only isolate already-active flow ticking. New target for this smoke scenario: exactly 12 combat characters, exactly 5,000 equipped entry point items across those characters, exactly 5,000 initially active flow commands, and a measured run representing 10 real-time seconds at 100 combat ticks per real-time second. The first threshold should be intentionally ambitious, around 10 ms average per tick, as a catastrophic regression guard rather than a final benchmark.

The test is allowed to let entry points create new flows during measurement because that better reflects a busy fight. To keep the setup deterministic, the test should still assert the exact character count, exact placed item count, exact initial active/created flow counts, and the expected created flow count after the measured trigger waves.

Previous inventory sizing mechanism: `InventoryAggregate.create` always created a fixed 17x8 inventory. This made the stress test misleading, because the constants in the test looked like scenario scale even though 1,000+ items required silently adding many characters. First replacement made inventory dimensions globally configurable through DI, but that still implied every character had the same inventory size.

Current mechanism: global inventory config now describes only the maximum allowed grid size through `InventoryGridConfiguration`. The actual inventory size is a character characteristic carried by `CreateCombatCharacterCommand` as a `GridDimensions` value. This keeps the current 17x8 character inventories explicit while allowing stress-test characters to use 100x50 and leaving room for future character-specific/GUI-determined inventory layouts.

Previous battle tick iteration mechanism: `BattleRuntime.tick` copied all combat characters with `ToList()` every tick before iterating. New mechanism: it iterates the current `IReadOnlyCollection<ICombatCharacterFacade>` directly, avoiding that per-tick list allocation. This assumes character membership is stable during the tick; if combat can later add/remove characters mid-tick, the combat runtime should introduce an explicit reusable participant snapshot rather than LINQ materialization.

GUI performance testing is a separate concern. A later PlayMode or scene-level test can include rendered inventory/UI, but this domain smoke should stay focused on combat runtime cost so failures point to tick/flow/item logic rather than prefabs, layout, rendering, or editor UI overhead.

Correction after first stress test run: active flow count dropped from 5,000 to 4,986 before measurement because the warmup ticked already-created dense-grid flows. With single-cell entry points packed next to each other, `GridAdjacencyRouter` performs random neighbor walking and some flows can hit a local dead end quickly. The test now warms up the session before manually creating the initial 5,000 flows, so the measured setup starts deterministically with exactly 5,000 active flows while still allowing flow completion and new flow creation during measurement.

AI 2026-05-27 - Flow processor settings validation

Previous mechanism: `FlowProcessorSettings` silently clamped `maxStepsPerSlice < 1` to `1` and had a placeholder `// verify` comment. The likely intent was to protect flow ticking from a zero/negative per-tick work budget, because `FlowProcessor.tick` only advances while `stepsProcessed < maxStepsPerSlice`.

New mechanism: keep `maxStepsPerSlice` as explicit flow configuration, supplied by DI defaults or test overrides, but make invalid values fail fast with `ArgumentOutOfRangeException`. This avoids hiding bad installer/test configuration and keeps the setting object as a small validated value object rather than moving the guard into hot-path flow code.

Follow-up interpretation: the old default value `64` on `FlowProcessorSettings` was a magic number and likely came from the previous async/yielded flow loop. In the ticked runtime model, pacing should eventually come from combat ticks plus item cast time/cooldown/duration, not from "process up to N nodes immediately." `maxStepsPerSlice` remains only as a per-flow, per-tick safety/work cap that prevents one connected flow from walking an unbounded number of nodes in a single tick. The default value should therefore live in composition/game configuration, not inside the settings constructor. `FlowProcessorSettings` now requires an explicit positive value, and `FlowProcessor` no longer creates hidden fallback settings.

User prefers Java-like code shape for domain/configuration values: private fields plus explicit getters instead of C# auto-properties such as `public int value { get; }`. `FlowProcessorSettings` now stores `maxStepsPerSlice` in a private readonly field and exposes `getMaxStepsPerSlice()`. The default safety cap was raised to 5,000 so it acts as an extreme loop/graph-walk guard rather than a likely gameplay limiter for combos.

AI 2026-05-27 - Combat tick rate and speed multiplier interpretation

User clarified tick-rate semantics: a combat tick is the smallest game time unit, not a domain concept of "one game second." The default GUI/runtime execution speed should be treated as 10 combat ticks per real-time second. Faster game speed means executing more ticks per real-time second, e.g. x10 speed means 100 ticks per real-time second.

Avoid describing mechanics as "1 game second = 10 ticks" for now. Prefer saying that a rule lasts or triggers after N ticks, while GUI/runtime speed decides how many ticks are executed per real-time second. The performance smoke test now names this explicitly with `DefaultCombatTicksPerRealSecond = 10`, `StressSpeedMultiplier = 10`, and `StressCombatTicksPerRealSecond = 100`.

Follow-up implementation: the default runtime speed now lives in `BattleSessionSettings`, owned by the battle/session layer rather than the domain tick model or only a test constant. Default is 10 combat ticks per real-time second. `BattleUIManager` uses `BattleSessionSettings.getRealSecondsPerCombatTick()` for its coroutine wait, and the performance smoke test reads the default from `BattleSessionSettings`.

AI 2026-06-05 - Current follow-up context after compaction

The latest active design thread before compaction was about defensive systems:

- avoid pure flat damage reduction as the core defense because it is hard to balance
- avoid pure percent resistance/cap systems that become mandatory endgame defaults
- explore defensive entry points that generate guard/defense portions
- consider guard portions with rating, trigger conditions, priority, tags, durability/strain, expiry, and storage limits
- rapid small attacks may serve as guard-breaking/guard-draining pressure if balanced carefully
- guard readability likely needs tooltips showing concrete examples such as "against 60 damage blocks about 55"
- critical rule: one abstract stat can be manageable if the opposing side remains concrete; two abstract opposing stats quickly destroy readability
- attack flow should preserve concrete damage-like values whenever possible, rather than becoming generic abstract `Offensive Power`
- AI must actively challenge ideas for exploits, dominant strategies, readability loss, balance risks, and architectural drift

AI 2026-06-05 - Defensive incoming damage tests added

Added unit coverage for the current event-triggered defensive flow mechanism before the planned guard-system redesign.

The tests currently treat this behavior as the expected baseline:

- event-triggered defensive entry points do not tick by normal combat tick alone
- `IncomingAttackDamageCombatEvent` can create a `FlowKind.Defense` flow before HP changes
- defensive flow resolution applies resolved damage directly and must not republish another incoming attack damage event
- if no defensive entry point exists, incoming attack damage applies directly
- if the first defensive entry point is busy, the character can use the next available event-triggered defensive entry point
- catalog `DefenseEntryPointGem + Shield` can process incoming damage through a defensive flow

Added a performance smoke for the current implementation: 12 characters, 800 offensive entry points, 1600 event defensive entry points, 120 measured ticks. The second verified run produced 47,200 incoming damage events, 47,200 defensive flows, 95,200 total created flows, and about 158.52 ms/tick. This is a regression canary for the current implementation, not a desired final combat-tick target.

AI 2026-06-05 - Combat production profile and defensive entry-point cursor

Optimized the current event-triggered defensive flow implementation after the defensive stress test exposed about 158.9 ms/tick.

Implemented:

- `CombatRuntimeSettings` with `Developer` and `Production` profiles
- combat hot-path logs are guarded by the combat runtime profile before interpolated log strings are built
- performance/stress tests use the production combat runtime profile
- event-triggered defensive entry points use a round-robin cursor instead of restarting every incoming damage event from entry point index 0

Verified defensive stress with the same 12 characters, 800 offensive entry points, 1600 event defensive entry points, and 120 measured ticks: 47,200 incoming damage events, 47,200 defensive flows, 95,200 total created flows, and about 20.27 ms/tick after the optimization. The test threshold was tightened to 25 ms/tick as a regression canary, because the current expected target is around 20 ms/tick.

AI 2026-06-05 - Round-robin defensive event entry point caveat

Current event-triggered defensive entry point processing uses a round-robin cursor when choosing among registered defensive event entry points. This was introduced as a hot-path optimization and avoids repeatedly scanning from entry point index 0.

This is not yet confirmed as the final gameplay rule. It may affect player-facing behavior because multiple available defensive entry points can alternate instead of always using inventory/registration priority. Revisit whether the final model should use round-robin, strict placement priority, explicit priority, conditions/tags, or another readable selection rule.

AI 2026-06-05 - Review of pending defensive event implementation

Reviewed the pending implementation for readability, safety, and architectural fit. The current direction still fits the modular-domain approach, but the implementation should keep two explicit safeguards:

- production/developer combat runtime profile must be configurable in real scene composition, not only in tests, so hot-path logs can be disabled for production-like runs
- round-robin defensive event entry point selection must stay explicit and tested, because it is a gameplay-affecting temporary choice rather than a final defense-selection rule

Remaining follow-ups to challenge later: event-triggered entry points still use a broad `CombatEvent` trigger kind instead of an explicit incoming-attack-damage trigger identity, and the registered defensive entry point index will need removal/rebuild handling when inventory removal or unequip becomes part of the combat lifecycle.

AI 2026-06-05 - CombatEvent and CombatHook terminology enforcement

Clarified that `CombatHook` is not a second kind of `CombatEvent`. `CombatEvent` is the hidden/synchronous domain fact; `CombatHook` is a game-mechanic reaction identity that can be player-facing, usually in `On...` language.

Implemented terminology guard in code:

- `CombatEvent` now implements `ICombatEvent`
- `CombatHook` / `ICombatHook` define player-facing hook identity and matching against combat fact types
- `IEntryPointDefinition`, placed entry points, and equipped entry points expose `getCombatHook()`
- combat-tick entry points declare `CombatHook.none()`
- event-triggered entry points must declare a concrete hook
- defensive incoming damage registration now checks `OnIncomingAttackDamage` instead of only checking broad `EntryPointTriggerKind.CombatEvent`

The technical `CombatContextEventHub` / DTO event area still exists separately for UI/context notifications and should not be confused with gameplay `CombatEvent` facts.

User clarified that domain/gameplay constructors should use existing `NullGuard` helpers and fail fast. Do not silently hide invalid inputs with fallbacks like `value ?? string.Empty`. AI 2026-06-06 - Guard power first implementation

User asked for a guard/defense system where entry points can produce `GuardPower` as a flow result and where prepared guards are visible in inventory UI as compact shield-like buff icons.

Implemented the first foundation:

- `GuardPower` is a separate flow output, not a `DamageRole`
- `AddGuardPower` lets entry points and normal items add guard power to a flow
- consuming a flow creates one prepared guard portion on the flow owner before attack/damage handling
- prepared guards live as `CombatCharacter` runtime state
- incoming `ResolvedDamage` is reduced by prepared guards before HP is changed
- each guard portion can react once per damage packet and is strained by the blocked amount
- current formula is a soft rating-style mitigation: `blocked = damage * (1 - (damage / (damage + guardPower))^2)`, rounded down and clamped so positive damage never becomes perfect zero damage
- current storage limit is 20 prepared guard portions; overflow merges into the weakest guard at 70% efficiency
- catalog `DefenseEntryPointGem` and `Shield` now demonstrate guard generation instead of flat `ATTACK` reduction
- inventory UI now renders compact prepared-guard icons near the top of inventory, with short labels such as `2.5k` and hover tooltips showing concrete block examples

Critical caveats preserved:

- this is a first generic guard foundation, not final defense balance
- there are no guard tags, typed wards, expiry, priority, or heavy-guard minimum-trigger rules yet
- the 20-portion cap and 70% overflow efficiency are provisional and need deterministic balance simulations against burst, rapid small hits, baiting, and late-game stacking
- the UI icon uses a free/runtime glyph-style approach for now and can be replaced by a prefab/free icon asset later

AI 2026-06-06 - Guard feedback, flow-end beams, and attack-scale guard values

User reported that guards did not visibly look like they were being depleted, guard icons appeared on top of the inventory instead of above it, and guard values such as 70/50 were too high relative to current attack values.

Previous mechanism:

- prepared guards mechanically reduced incoming `ResolvedDamage`, strained by blocked damage, and disappeared when empty
- UI only refreshed the final prepared-guard list, so a guard created and consumed during the same defensive flow could appear to "not work" or disappear without an explanatory visual
- flow consumption did not expose the final processed item/slot, so UI could not draw a reliable end-of-flow beam

Implemented follow-up:

- flow consumption now carries the final `ItemFlowProcessingSlot`
- combat-context DTO events announce guard creation and attack creation with the final flow slot
- character DTO events announce guard absorption with incoming damage, blocked damage, remaining damage, destroyed guard count, and remaining guard power
- UI keeps domain separation by translating DTO facts into beams/popups: final item to guard icon, final item to target character, and a `G -blocked / remaining` popup on guard absorption
- prepared guard icons are positioned above the inventory area, not inside the inventory field
- catalog `DefenseEntryPointGem` and `Shield` guard values were lowered to attack-scale values: 2 and 5

Added tests:

- guard creation event includes the final flow item slot
- attack creation event includes the final flow item slot and concrete target
- guard absorption event proves blocked/pass-through values and guard destruction
- prepared guard overlay is positioned above the inventory layer

Verified with Unity EditMode test run: 486/486 passing on 2026-06-06.

AI 2026-06-07 - Port-aware outputs and readable guard queue

User challenged the guard portion system because "add defense to the weakest guard" and similar behavior are hidden global rules. The desired direction is fewer broad rules and more behavior controlled by item/output characteristics. User also identified that item-by-item guard creation can create too many small guards and make the inventory unreadable.

Design discussion outcome:

- output items/ports can become the watched "in/out" points for flows
- entry/input should mostly answer when/how a flow starts, not hard-code attack/defense semantics
- output should be a commit point for accumulated flow power
- ports should be visually marked and explain what they react to / what they do, but UI must not own gameplay rules
- flow that does not reach an output should communicate that fact
- guard readability and balance pull against each other: unlimited guard portions are unreadable, but hard caps can create balance traps

Implemented prototype direction:

- added item port metadata and example `PulseInputPort` / `BasicOutputPort`
- kept old entry-point logic for compatibility
- output ports commit accumulated attack/guard power without casting or executing their own effects
- port-aware input flows that miss output are discarded and reported
- legacy no-output flows still commit but report a no-output fact
- UI shows `IN`/`OUT` badges, input/output/no-output signals, and output-origin beams
- item add buttons now map to concrete item definitions instead of random item pools
- guard storage is now max 30 portions
- guard overflow replaces the oldest guard instead of merging into the weakest
- incoming damage consumes guards left-to-right instead of choosing the strongest
- replacement and absorption carry enough data for UI to show push/split feedback

Critical caveats:

- `PulseInputPort` is still an adapter over the existing `IEntryPointDefinition`/`FlowKind` system, so it is not the final neutral input architecture
- future port-aware defensive inputs need a specific no-output rule, because the current defensive event processor stops direct damage as soon as a defensive flow is created
- future work should let output/item characteristics define policies such as split guard into N portions, replace oldest/weakest, fill free slots only, or discard overflow
- full EditMode verification was blocked in this session because Unity batchmode could not open the project while another Unity instance had it open

AI 2026-06-22 - Battle time control

Implemented runtime battle speed control as a scheduling layer over `CombatTicks`, not as a mutation of combat-domain values.

Current behavior:

- `BattleTimeController` converts unscaled real elapsed time into ready combat ticks
- supported speed range is 0.2x to 5x
- pause produces no combat ticks and does not accumulate delayed ticks while paused
- `BattleUIManager` now drives battle ticks from `Update()` with `Time.unscaledDeltaTime` instead of a fixed `WaitForSeconds` coroutine
- battle UI creates a compact runtime time-control panel with speed up/down, pause/resume, a vertical speed slider, and a short status label

Critical design point:

- changing battle speed changes only when ticks are executed in real time; it must not change item cast ticks, trigger intervals, damage, guard power, or other deterministic combat logic

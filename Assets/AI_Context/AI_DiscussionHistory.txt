AI 2026-06-05 - Round-robin defensive event entry point caveat

Current event-triggered defensive entry point processing uses a round-robin cursor when choosing among registered defensive event entry points. This was introduced as a hot-path optimization and avoids repeatedly scanning from entry point index 0.

This is not yet confirmed as the final gameplay rule. It may affect player-facing behavior because multiple available defensive entry points can alternate instead of always using inventory/registration priority. Revisit whether the final model should use round-robin, strict placement priority, explicit priority, conditions/tags, or another readable selection rule.

AI 2026-06-06 - Guard power first implementation

User asked for a guard/defense system where entry points can produce `GuardPower` as a flow result and where prepared guards are visible in inventory UI as compact shield-like buff icons.

Implemented the first foundation:

- `GuardPower` is a separate flow output, not a `DamageRole`
- `AddGuardPower` lets entry points and normal items add guard power to a flow
- consuming a flow creates one prepared guard portion on the flow owner before attack/damage handling
- prepared guards live as `CombatCharacter` runtime state
- incoming `ResolvedDamage` is reduced by prepared guards before HP is changed
- each guard portion can react once per damage packet and is strained by the blocked amount
- current formula is a soft rating-style mitigation: `blocked = damage * (1 - (damage / (damage + guardPower))^2)`, rounded down and clamped so positive damage never becomes perfect zero damage
- current storage limit is 20 prepared guard portions; overflow merges into the weakest guard at 70% efficiency
- catalog `DefenseEntryPointGem` and `Shield` now demonstrate guard generation instead of flat `ATTACK` reduction
- inventory UI now renders compact prepared-guard icons near the top of inventory, with short labels such as `2.5k` and hover tooltips showing concrete block examples

Critical caveats preserved:

- this is a first generic guard foundation, not final defense balance
- there are no guard tags, typed wards, expiry, priority, or heavy-guard minimum-trigger rules yet
- the 20-portion cap and 70% overflow efficiency are provisional and need deterministic balance simulations against burst, rapid small hits, baiting, and late-game stacking
- the UI icon uses a free/runtime glyph-style approach for now and can be replaced by a prefab/free icon asset later

AI 2026-06-07 - Port-aware outputs and readable guard queue

User challenged the guard portion system because "add defense to the weakest guard" and similar behavior are hidden global rules. The desired direction is fewer broad rules and more behavior controlled by item/output characteristics. User also identified that item-by-item guard creation can create too many small guards and make the inventory unreadable.

Design discussion outcome:

- output items/ports can become the watched "in/out" points for flows
- entry/input should mostly answer when/how a flow starts, not hard-code attack/defense semantics
- output should be a commit point for accumulated flow power
- ports should be visually marked and explain what they react to / what they do, but UI must not own gameplay rules
- flow that does not reach an output should communicate that fact
- guard readability and balance pull against each other: unlimited guard portions are unreadable, but hard caps can create balance traps

Implemented prototype direction:

- added item port metadata and example `PulseInputPort` / `BasicOutputPort`
- kept old entry-point logic for compatibility
- output ports commit accumulated attack/guard power without casting or executing their own effects
- port-aware input flows that miss output are discarded and reported
- legacy no-output flows still commit but report a no-output fact
- UI shows `IN`/`OUT` badges, input/output/no-output signals, and output-origin beams
- item add buttons now map to concrete item definitions instead of random item pools
- guard storage is now max 30 portions
- guard overflow replaces the oldest guard instead of merging into the weakest
- incoming damage consumes guards left-to-right instead of choosing the strongest
- replacement and absorption carry enough data for UI to show push/split feedback

Critical caveats:

- `PulseInputPort` is still an adapter over the existing `IEntryPointDefinition`/`FlowKind` system, so it is not the final neutral input architecture
- future port-aware defensive inputs need a specific no-output rule, because the current defensive event processor stops direct damage as soon as a defensive flow is created
- future work should let output/item characteristics define policies such as split guard into N portions, replace oldest/weakest, fill free slots only, or discard overflow

AI 2026-06-22 - Defense layer order, base armor/resistance, and future barrier

User is exploring a readable global armor/resistance-like defense value as an additional layer beside prepared guards.

Current design intent:

- every battle may start from a baseline defensive value around 1000
- defensive flows may generate more of this value during combat
- combat should pressure the player to maintain this value; no armor generation means incoming attacks become increasingly dangerous as the value is strained down
- generating above the baseline should be meaningfully rewarded, but the percentage mitigation should have diminishing returns
- investing only in this value should not be automatically best because attacks may pierce, ignore, drain, break, or otherwise counter the layer
- guard remains useful as a generated/tactical layer and may protect or interact with this base defense, rather than replacing it

UI/readability direction:

- defense layers may be shown as ordered rows above the item grid
- the highest row is processed first
- near-term rows are the global armor/resistance-like value and prepared guards
- default near-term order is armor/resistance first, then guard, but this is still design exploration
- a later equipment-editing feature may allow changing the processing order of defense layers, such as armor before guard or guard before armor
- row order should be visible and inspectable, not an invisible combat-pipeline rule

Naming constraint:

- user is also planning a future barrier layer as generated extra HP or a similar buffer
- therefore the armor/resistance-like layer should not be named `Barrier`
- `Armor` currently sounds too physical/narrow for this global defense concept, so better naming remains open

AI 2026-06-22 - Implemented Stability defense layer

Implemented the first version of the readable global armor/resistance-like layer under the name `Stability`.

Previous incoming damage behavior:

- resolved incoming damage was processed by prepared guards first, then remaining damage went to HP
- there was no baseline global defensive value that every combat character started from

Current implemented behavior:

- every combat character starts each battle with baseline Stability 1000
- incoming damage is processed in the visible order Stability -> prepared Guard -> HP

AI 2026-06-27 - Stability baseline reference and overcap decay

User identified that absolute Stability values and direct damage-scaled strain can make late-game Stability swing too hard or turn anti-Stability stats into mandatory offense. The design shifted away from treating `1000 Stability = stronger percentage reduction` as an absolute global curve.

Previous implemented Stability behavior:

- default baseline Stability was 1000
- mitigation used an absolute reference formula: `damageAfter = incomingDamage * 1000 / (1000 + currentStability)`
- ordinary incoming attacks strained Stability by roughly one quarter of incoming damage
- more absolute Stability always moved directly along the same global reduction curve

New implemented Stability behavior:

- default baseline Stability is now 100
- mitigation is relative to the character's baseline: `damageAfter = incomingDamage * baselineStability / (baselineStability + currentStability)`
- full baseline Stability gives about 50% reduction whether the baseline is 100 or a future late-game value such as 1000 or 2000
- ordinary incoming attacks now strain Stability by roughly one third of incoming damage, clamped at 0 for ordinary strain
- baseline Stability is now the size/reference of the Stability bar: higher baseline means the character maintains its standard reduction longer under strain, rather than starting with a higher percent cap
- Stability can exceed baseline, but over-baseline Stability decays during combat ticks
- over-baseline decay currently does not start until about 130% baseline; it becomes noticeable around 200% baseline and very fast around 300% baseline

Design reason:

- this preserves a readable single Stability value while scaling across early/late game
- it avoids a mandatory offensive `Pressure` stat that permanently weakens all future hits
- it rewards defensive generation above baseline but makes extreme overcap temporary, so pure Stability stacking should not become the obvious default

AI 2026-06-27 - Slower steerable Stability overcap decay

User pointed out that the first over-baseline Stability decay was far too fast for a baseline of 100, especially because losing a minimum of 1 Stability per tick near the threshold made the overcap feel harsh instead of gently temporary.

Implemented follow-up:

- over-baseline decay now starts at about 110% baseline instead of 130%
- decay near the threshold is much slower because it accumulates fractional decay units before removing a full Stability point
- the current curve is convex, effectively the opposite of logarithmic for gameplay purposes: it starts very slowly and accelerates as Stability moves farther above the decay threshold
- current prototype tuning for baseline 100: around 130 Stability does not lose a full point after one tick, around 200 loses a few points per tick, and around 300 loses much faster
- this curve should be treated as future character/balance space; later stats/items may tune the threshold, retention, or curve strength instead of making the global rule fixed forever

Design reason:

- small overcap should feel like a useful buffer, not immediately wasted
- extreme overcap should still be temporary so pure Stability stacking does not become the obvious default
- fractional decay avoids making low baseline values behave too harshly because of integer rounding

AI 2026-06-27 - Ticked damage packets through defense layers

User wanted damage to be readable as a falling number that travels from the source character and passes through defense layers: incoming value, Stability output, Guard output, then HP. The timing must be real combat timing, not only animation, because offscreen characters and future `OnHit`-style hooks need the same processing delay.

Previous behavior:

- offensive flow output immediately dispatched `IncomingAttackDamageCombatEvent`
- if no defensive flow intercepted, incoming damage was immediately applied through Stability -> Guard -> HP in one synchronous call
- defensive flow output also applied resolved damage immediately through all layers

New implemented behavior:

- offensive attack output creates a ticked `ActiveDamagePacket`
- default packet stage duration is currently 10 combat ticks and is configured through `DamagePacketProcessingSettings`
- packet stages are Travel -> IncomingHit -> Stability -> Guard -> HP
- IncomingHit dispatches the existing `IncomingAttackDamageCombatEvent`, preserving event-triggered defensive flow behavior
- if no defensive flow handles the hit, CombatContext enqueues resolved damage back into the packet pipeline
- defensive flow output also enqueues resolved damage into the same Stability -> Guard -> HP packet pipeline instead of applying it immediately
- `CombatContext.combatTick(...)` owns packet progression, so visible and offscreen characters process damage timing identically
- `BattleRuntime` ticks existing combat-context packets before ticking character flow/item logic for the frame
- layer processing publishes neutral `DamagePacketLayerProcessedDtoEvent` facts for UI/testing
- the current UI uses these facts for simple staged popups near the target character; this is a first visualization, not the final polished falling-number path over inventory layer rows

Testing notes:

- added `DamagePacketProcessingTest` covering configurable packet stage duration and Travel -> IncomingHit -> Stability -> Guard -> HP sequencing
- updated existing damage/guard/stability tests to advance combat-context packet ticks explicitly when they assert final HP or layer absorption
- verified EditMode tests: 502/502 passing
